## 各章节详细说明

### 头部信息

**任务名称**:
- 从用户消息提取
- 简洁明确,通常 5-10 个字

**创建时间**:
- Session 创建时的时间戳
- 格式: YYYY-MM-DD HH:mm
- 示例: 2026-03-09 14:30

**最后更新**:
- 每次更新 INDEX.md 时更新此时间戳
- 用于判断 session 活跃度

### 任务概览

**内容**:
- 简洁描述任务目标
- 最多 5 行
- 回答"要做什么"

**示例**:
```markdown
## 📋 任务概览

为系统添加用户登录功能,包括:
- 用户名/密码登录
- JWT token 生成和验证
- 登录状态管理
- 登出功能
```

### 需求背景

**内容**:
- 说明为什么做这个任务
- 最多 3 句话
- 回答"为什么要做"

**示例**:
```markdown
## 🎯 需求背景

当前系统没有用户认证机制,所有功能都是开放的。
需要添加登录功能来保护敏感数据和操作。
这是产品路线图中 Q1 的重要功能。
```

### 分析主线

**内容**:
- 核心分析思路
- 最多 5 个要点
- 保持高层次,不陷入细节

**示例**:
```markdown
## 🔍 分析主线

1. 技术方案选择 - 使用 JWT token,无需 session 存储
2. 数据模型设计 - 用户表需要增加 password_hash 字段
3. API 端点设计 - POST /api/auth/login 和 /api/auth/logout
4. 前端集成 - 使用 Context API 管理登录状态
5. 安全考虑 - 密码加密、token 过期、HTTPS 传输
```

### 文档关系图

**内容**:
- 列出所有相关文档
- 使用相对路径链接
- 最多 10 个文档
- 简短说明每个文档的用途

**示例**:
```markdown
## 📂 文档关系图

- [INDEX.md](INDEX.md) - 主线文档(本文档)
- [requirements-alignment.md](requirements-alignment.md) - 需求对齐
- [design.md](design.md) - 技术设计
- [impact-analysis.md](impact-analysis.md) - 影响面分析
- [implementation-plan.md](implementation-plan.md) - 实现计划
- [progress-details.md](progress-details.md) - 详细进度
```

### 当前进度

**总体进度**:
- 百分比形式: X%
- 完成项/总项: 已完成 Y/Z 项

**关键里程碑**:
- 最多 5 个
- 使用 checkbox 格式: `- [x]` 已完成, `- [ ]` 未完成
- 按时间顺序排列

**示例**:
```markdown
## 📊 当前进度

**总体进度**: 60% (已完成 3/5 项)

**关键里程碑**:
- [x] 需求对齐完成
- [x] 设计完成
- [x] 后端 API 实现完成
- [ ] 前端集成
- [ ] 测试和部署

**详细进度**: 见 [progress-details.md](progress-details.md)
```

### 关键决策

**内容**:
- 记录重要技术决策
- 最多 5 个
- 每个决策 1-2 句话
- 简短说明原因

**示例**:
```markdown
## ✅ 关键决策

1. **使用 JWT token** - 无需服务端存储 session,更适合分布式架构
2. **密码使用 bcrypt 加密** - 业界标准,安全性高
3. **Token 有效期 24 小时** - 平衡安全性和用户体验
4. **不支持第三方登录** - MVP 阶段先实现基本功能
5. **前端使用 Context API** - 简单轻量,满足当前需求
```

### 相关资源

**代码文件**:
- 列出主要改动的文件
- 使用相对路径
- 最多 10 个文件
- 简短说明每个文件的用途

**参考文档**:
- 相关 Issue、PR、设计文档等
- 最多 5 个参考
- 使用 markdown 链接格式

**示例**:
```markdown
## 🔗 相关资源

**代码文件**:
- `src/auth/login.ts` - 登录逻辑
- `src/auth/jwt.ts` - JWT 生成和验证
- `src/models/user.ts` - 用户模型
- `tests/auth.test.ts` - 认证测试

**参考文档**:
- [Issue #123: 添加用户认证](https://example.com/issues/123)
- [JWT 最佳实践](https://jwt.io/introduction)
```

## 内容限制规则

### 为什么要限制内容?

**原因 1: 避免信息过载**
- INDEX.md 是导航中心,不是详细记录
- 过多内容会淹没核心信息

**原因 2: 保持可读性**
- 用户应该能快速浏览 INDEX.md
- 获取任务整体情况

**原因 3: 详细内容分离**
- 详细内容放在独立文档中
- 通过链接访问

### 具体限制

| 章节 | 限制 | 原因 |
|------|------|------|
| 任务概览 | 最多 5 行 | 保持简洁 |
| 需求背景 | 最多 3 句话 | 只说明核心原因 |
| 分析主线 | 最多 5 个要点 | 高层次思路 |
| 文档关系图 | 最多 10 个文档 | 只列核心文档 |
| 关键里程碑 | 最多 5 个 | 重要里程碑 |
| 关键决策 | 最多 5 个 | 重要决策 |
| 相关资源 | 最多 15 个 | 核心资源 |

### 详细内容分离

**详细进度** → `progress-details.md`:
- 每个步骤的详细状态
- 遇到的问题和解决方案
- 时间记录

**详细记录** → `detailed-records.md`:
- 完整的分析过程
- 所有讨论和决策细节
- 历史变更记录

**改动总结** → `changes-summary.md`:
- 所有改动的文件列表
- 每个文件的改动说明
- 测试结果

**错误日志** → `error-recovery.md`:
- 遇到的错误
- 排查过程
- 解决方案

## 更新规则

### 何时更新 INDEX.md

**必须更新**:
1. 完成关键里程碑
2. 做出重要技术决策
3. 创建新的文档(更新文档关系图)
4. 任务进度有显著变化(≥10%)

**可以更新**:
- 增加新的相关资源
- 补充需求背景
- 调整分析主线

**不需要更新**:
- 小的代码改动
- 调试和测试过程中的临时改动
- 不影响整体进度的细节调整

### 更新内容

**必须更新**:
- "最后更新"时间戳(每次更新都要更新)
- 进度百分比和里程碑状态
- 文档关系图(如果创建了新文档)

**可能更新**:
- 关键决策(如果有新的重要决策)
- 相关资源(如果有新的核心文件或参考)

**通常不更新**:
- 任务概览(除非任务目标改变)
- 需求背景(除非理解有变化)
- 分析主线(除非思路有重大调整)

## 最佳实践

### 1. 保持简洁

**✅ 好的做法**:
- 任务概览 5 行以内
- 每个章节只包含核心信息
- 详细内容分离到独立文档

**❌ 不好的做法**:
- 任务概览写了 20 行
- 关键决策列出 15 个
- 把所有详细内容都放在 INDEX.md

### 2. 及时更新

**✅ 好的做法**:
- 完成里程碑立即更新
- 做出重要决策立即记录
- 更新时间戳

**❌ 不好的做法**:
- 任务完成了,进度还是 0%
- 做了重要决策,没有记录
- 时间戳一直不更新

### 3. 使用双向链接

**✅ 好的做法**:
- INDEX.md 链接到所有相关文档
- 每个文档开头链接回 INDEX.md
- 形成清晰的文档导航网络

**❌ 不好的做法**:
- INDEX.md 不链接其他文档
- 其他文档也不链接 INDEX.md
- 文档之间孤立
