# 文档清理策略

> 防止 temp/sessions/ 目录膨胀的清理和归档机制

## 为什么需要清理?

### 问题

**temp/sessions/ 目录膨胀**:
- 随着时间推移,session 越来越多
- 占用磁盘空间
- 难以找到最近的 session

**示例**:
```
temp/sessions/
├── 2026-01-10-0900-fix-bug/         # 2 个月前
├── 2026-01-15-1400-add-feature/     # 2 个月前
├── ... (50+ 个旧 sessions)
├── 2026-03-08-1000-refactor/        # 昨天
└── 2026-03-09-1430-add-auth/        # 今天
```

### 解决方案

**定期清理**:
- 归档重要 session
- 删除临时 session
- 保持目录整洁

---

## 清理时机

### 触发条件

1. **Session 完成**: 任务完成、代码已提交
2. **Session 过多**: `temp/sessions/` 下超过 20 个目录
3. **用户请求**: 用户明确要求清理

### 自动提示

**Session 完成时**:
```
✅ 任务已完成,代码已提交

📁 Session: temp/sessions/2026-03-09-1430-add-auth/

是否清理此 session?
1. 归档 - 移动到 archive/ 目录保留
2. 删除 - 直接删除(不推荐,除非是简单任务)
3. 保留 - 暂时保留在当前位置
```

**Session 过多时**:
```
⚠️ 发现 temp/sessions/ 下有 25 个目录,建议清理

最旧的 sessions:
1. 2026-01-10-0900-fix-bug (2 个月前)
2. 2026-01-15-1400-add-feature (2 个月前)
...

是否开始清理?
```

## 清理策略

### 重要 Session → 归档

**归档条件**:
- 任务持续时间 > 1 天
- 包含重要决策和设计文档
- 用户明确标记为重要

**归档位置**: `temp/sessions/archive/YYYY-MM/`

**示例**:
```
temp/sessions/archive/
├── 2026-01/
│   ├── 2026-01-10-0900-fix-bug/
│   └── 2026-01-15-1400-add-feature/
├── 2026-02/
│   └── 2026-02-20-1000-refactor/
└── 2026-03/
    └── 2026-03-09-1430-add-auth/
```

### 临时 Session → 删除

**删除条件**:
- 任务持续时间 < 1 小时
- 只是简单的 bug 修复或小改动
- 没有重要文档

**直接删除**: 不保留,释放空间

### 活跃 Session → 保留

**保留条件**:
- 最近 7 天内有更新
- 任务尚未完成
- 用户明确要求保留

---

## 归档流程

### 步骤 1: 识别需要归档的 Session

**自动识别**:
- 任务完成(进度 100%)
- 代码已提交
- 包含重要文档

**用户确认**:
```
📁 Session: temp/sessions/2026-03-09-1430-add-auth/

此 session 包含以下文档:
- INDEX.md
- requirements-alignment.md
- design.md
- impact-analysis.md
- implementation-plan.md

是否归档?
```

### 步骤 2: 创建归档目录

```bash
mkdir -p temp/sessions/archive/2026-03
```

### 步骤 3: 移动 Session

```bash
mv temp/sessions/2026-03-09-1430-add-auth/ \
   temp/sessions/archive/2026-03/
```

### 步骤 4: 通知用户

```
✅ Session 已归档到: temp/sessions/archive/2026-03/2026-03-09-1430-add-auth/
```

---

## 删除流程

### 步骤 1: 识别临时 Session

**自动识别**:
- 任务持续时间 < 1 小时
- 没有重要文档(只有 INDEX.md 或少量文档)
- 简单任务(如 bug 修复)

**用户确认**:
```
📁 Session: temp/sessions/2026-03-08-0900-fix-typo/

此 session 是简单任务,建议删除

是否删除?
```

### 步骤 2: 删除 Session

```bash
rm -rf temp/sessions/2026-03-08-0900-fix-typo/
```

### 步骤 3: 通知用户

```
✅ Session 已删除: temp/sessions/2026-03-08-0900-fix-typo/
```

## 批量清理

### 清理流程

```
[扫描所有 sessions] → [分类] → [生成清理清单] → [用户确认] → [执行清理]
```

### 清理清单

```markdown
# 清理清单

## 建议归档 (5 个)

1. 2026-01-10-0900-major-refactor (2 个月前, 包含重要设计)
2. 2026-01-15-1400-add-feature (2 个月前, 包含 API 设计)
3. 2026-02-01-1000-database-migration (1 个月前, 包含迁移脚本)
4. 2026-02-15-1400-performance-optimization (3 周前, 包含性能分析)
5. 2026-03-01-0900-security-fix (1 周前, 包含安全分析)

## 建议删除 (10 个)

1. 2026-01-05-0900-fix-typo (2 个月前, 简单修复)
2. 2026-01-12-1000-update-config (2 个月前, 配置更新)
...

## 保留 (3 个)

1. 2026-03-08-1000-refactor (昨天, 进行中)
2. 2026-03-09-1430-add-auth (今天, 进行中)
3. 2026-03-09-1500-fix-bug (今天, 已完成但刚完成)

## 总结

- 归档: 5 个
- 删除: 10 个
- 保留: 3 个
- **清理后剩余**: 3 个活跃 sessions + 归档目录

是否继续清理?
```

## 自动清理规则

### 规则 1: 自动归档

**条件**:
- Session 完成超过 7 天
- 包含 ≥5 个文档
- 任务持续时间 > 1 天

**操作**: 自动移动到 archive/ 目录,无需确认

### 规则 2: 自动删除

**条件**:
- Session 完成超过 30 天
- 只有 INDEX.md 或少量文档
- 任务持续时间 < 1 小时

**操作**: 自动删除,但提前通知用户

### 规则 3: 保留

**条件**:
- 最近 7 天内有更新
- 任务尚未完成

**操作**: 保留,不清理

## 用户确认

### 确认界面

```
📁 清理 temp/sessions/ 目录

发现以下 sessions 需要清理:

【归档】(保留在 archive/ 目录):
✓ 2026-01-10-0900-major-refactor
✓ 2026-01-15-1400-add-feature

【删除】(释放空间):
✓ 2026-01-05-0900-fix-typo
✓ 2026-01-12-1000-update-config

【保留】(暂不处理):
  2026-03-08-1000-refactor
  2026-03-09-1430-add-auth

是否继续? (Y/n)
```

### 选择性清理

**用户可以调整**:
- 取消某个 session 的归档/删除
- 将"删除"改为"归档"
- 将"归档"改为"保留"

## 最佳实践

### 1. 及时清理

**✅ 好的做法**:
- Session 完成后及时清理
- 定期(每周)检查 sessions 目录
- 归档重要 session,删除临时 session

**❌ 不好的做法**:
- 从不清理,积累大量旧 sessions
- 所有 sessions 都保留
- 所有 sessions 都删除(丢失重要文档)

### 2. 合理归档

**✅ 好的做法**:
- 重要 session 归档保留
- 按月分类(archive/YYYY-MM/)
- 保持归档目录结构清晰

**❌ 不好的做法**:
- 所有 session 都归档(占用空间)
- 归档目录混乱,难以查找
- 从不归档,直接删除

### 3. 用户确认

**✅ 好的做法**:
- 清理前询问用户
- 提供清理清单供用户确认
- 允许用户调整清理方案

**❌ 不好的做法**:
- 自动清理,不通知用户
- 不提供清单,用户不知道会删除什么
- 不允许用户调整
