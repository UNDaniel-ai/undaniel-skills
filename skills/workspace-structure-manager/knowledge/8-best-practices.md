## 常见场景处理

### 场景 1: 新任务开始

**用户消息**: "帮我添加用户登录功能"

**✅ 正确做法**:
1. 立即创建 session 目录: `temp/sessions/2026-03-09-1430-add-login/`
2. 创建 INDEX.md,提取任务目标
3. 创建 requirements-alignment.md,确认需求
4. 开始执行任务

**❌ 错误做法**:
- 在项目根目录创建 plan.md, design.md
- 不创建 session,直接开始写代码
- 等待用户说"创建 session"

---

### 场景 2: 继续之前的任务

**用户消息**: "继续之前的登录功能任务"

**✅ 正确做法**:
1. 识别最近的相关 session(通过关键词或时间)
2. 读取 INDEX.md 恢复上下文
3. 更新"最后更新"时间戳
4. 继续任务

**❌ 错误做法**:
- 创建新的 session(导致多个重复 session)
- 不读取 INDEX.md,丢失之前的进度
- 不更新时间戳

---

### 场景 3: 复杂任务分解

**用户消息**: "添加登录、权限管理和审计日志"

**✅ 正确做法**:
1. 创建主 session: `temp/sessions/2026-03-09-1430-add-auth/`
2. 在 INDEX.md 中列出 3 个子任务
3. 创建子任务目录: `sub-task-01-add-login/`, `sub-task-02-add-permissions/`, `sub-task-03-add-audit/`
4. 逐一执行子任务

**❌ 错误做法**:
- 创建 3 个独立的 session(关系不清晰)
- 不分解,一次性处理所有功能(过于复杂)
- 并行执行多个子任务(容易混乱)

---

### 场景 4: 提交代码前

**用户消息**: "准备提交代码"

**✅ 正确做法**:
1. 检查所有改动的项目(包括 submodules)
2. 在 INDEX.md 中记录改动清单
3. 提供提交前检查清单
4. 用户确认后再提交

**❌ 错误做法**:
- 直接提交,不检查其他项目
- 不记录改动清单
- 不提供检查清单,遗漏重要步骤

---

### 场景 5: Session 完成

**用户消息**: "任务完成了"

**✅ 正确做法**:
1. 更新 INDEX.md,进度改为 100%
2. 询问用户是否清理 session(归档/删除/保留)
3. 根据用户选择执行清理

**❌ 错误做法**:
- 不更新 INDEX.md
- 不提示清理,导致 sessions 目录膨胀
- 自动删除,没有询问用户

---

## 反模式和陷阱

### 反模式 1: 在项目根目录创建临时文档

**表现**:
```
project-root/
├── plan.md                    # ❌
├── design.md                  # ❌
├── progress.md                # ❌
└── src/
```

**问题**: 污染项目根目录,难以清理

**解决**: 始终在 `temp/sessions/{session}/` 下创建临时文档

---

### 反模式 2: INDEX.md 过于详细

**表现**:
```markdown
## 任务概览

[100 行的详细描述]

## 分析主线

[20 个要点]

## 关键决策

[15 个决策]
```

**问题**: 信息过载,失去导航作用

**解决**: 遵守内容限制规则,详细内容分离到独立文档

---

### 反模式 3: 多个任务共用一个 Session

**表现**:
```
temp/sessions/2026-03-09-1430-work/
├── INDEX.md           # 混合多个任务
├── login-design.md    # 任务 1
├── payment-plan.md    # 任务 2
└── report-bug.md      # 任务 3
```

**问题**: 文档混乱,难以追踪进度

**解决**: 每个任务独立 session

---

### 反模式 4: 从不清理 Session

**表现**:
```
temp/sessions/
├── ... (100+ 个旧 sessions)
```

**问题**: 占用空间,难以查找

**解决**: 及时清理,归档重要 session,删除临时 session

---

### 反模式 5: 不读取 INDEX.md

**表现**:
```
[用户说"继续之前的任务"]
[Agent 不读取 INDEX.md,不知道之前做了什么]
[从头开始,重复工作]
```

**问题**: 丢失上下文,重复工作

**解决**: 切换 session 时必须读取 INDEX.md

## 性能优化建议

### 优化 1: 减少文件数量

**问题**: 每个 session 创建过多文档

**优化**:
- 只创建必要的文档
- 合并相关文档(如 requirements + impact-analysis)
- 使用 progress-details.md 记录详细进度,而不是每个步骤一个文档

---

### 优化 2: 控制 INDEX.md 大小

**问题**: INDEX.md 过大,加载慢

**优化**:
- 遵守内容限制规则
- 详细内容分离到独立文档
- 定期审查并精简

---

### 优化 3: 定期清理

**问题**: sessions 目录膨胀,查找慢

**优化**:
- 每周清理一次
- 自动归档旧 sessions
- 保持活跃 sessions 数量在 10 个以内

---

## 与用户交互的最佳实践

### 1. 自动化,但透明

**✅ 好的做法**:
- 自动创建 session,但通知用户
- 自动更新 INDEX.md,但告知用户更新了什么

**示例**:
```
✅ Session 已创建: temp/sessions/2026-03-09-1430-add-login/
📄 INDEX.md 已初始化

正在创建需求对齐文档...
```

**❌ 不好的做法**:
- 悄悄创建 session,不通知用户
- 用户不知道发生了什么

---

### 2. 关键决策询问用户

**✅ 好的做法**:
- 清理 session 前询问用户
- 重要技术决策前征求用户意见

**示例**:
```
📁 Session: temp/sessions/2026-03-09-1430-add-login/

任务已完成,是否清理此 session?
1. 归档
2. 删除
3. 保留
```

**❌ 不好的做法**:
- 自动删除 session,不询问用户
- 不征求用户意见,直接做技术决策

---

### 3. 提供清晰的进度反馈

**✅ 好的做法**:
- 定期更新 INDEX.md 的进度
- 告知用户当前阶段和进度

**示例**:
```
📊 当前进度: 60%
✅ 需求对齐完成
✅ 设计完成
⏸️ 实现中 (60%)
⏳ 测试
⏳ 部署
```

**❌ 不好的做法**:
- 从不更新进度
- 用户不知道任务进展如何

## 故障排除指南

### 问题 1: 找不到之前的 Session

**症状**: 用户说"继续之前的任务",但找不到对应的 session

**可能原因**:
- Session 被删除
- Session 名称不匹配
- Session 在其他目录

**排查步骤**:
1. 列出所有 sessions: `ls -t temp/sessions/`
2. 检查 archive 目录: `ls -t temp/sessions/archive/*/`
3. 搜索关键词: `find temp/sessions -name "*keyword*"`

**解决方案**:
- 如果在 archive 中,从 archive 恢复
- 如果已删除,创建新 session
- 如果名称不匹配,手动选择正确的 session

---

### 问题 2: INDEX.md 内容混乱

**症状**: INDEX.md 内容过多、结构混乱

**可能原因**:
- 没有遵守内容限制规则
- 详细内容没有分离

**解决方案**:
1. 重新组织 INDEX.md,只保留核心信息
2. 将详细内容移到 progress-details.md, detailed-records.md 等
3. 更新文档关系图

---

### 问题 3: Session 目录过多

**症状**: `temp/sessions/` 下有 50+ 个 sessions,难以查找

**可能原因**:
- 从不清理
- 自动清理功能未启用

**解决方案**:
1. 执行批量清理
2. 归档重要 sessions
3. 删除临时 sessions
4. 启用自动清理规则

---

### 问题 4: 多项目改动遗漏

**症状**: 提交代码后发现有些项目没有提交

**可能原因**:
- 没有检查 submodules
- 没有记录改动清单

**解决方案**:
1. 提交前使用多项目追踪功能
2. 在 INDEX.md 中记录所有改动的项目
3. 使用提交前检查清单

---

### 问题 5: 子任务混乱

**症状**: 多个子任务并行,进度混乱

**可能原因**:
- 没有逐一执行子任务
- 没有更新子任务状态

**解决方案**:
1. 暂停所有子任务,选择一个继续
2. 更新 INDEX.md 的子任务状态
3. 逐一完成子任务

---

## 总结

### 核心原则

1. **自动化优先** - Agent 自动执行,无需用户手动管理
2. **保持简洁** - INDEX.md 只包含核心信息
3. **双向链接** - INDEX.md ↔ 其他文档相互引用
4. **灵活适配** - 尊重用户指定的路径
5. **及时清理** - 防止 `temp/sessions/` 膨胀

### 关键检查清单

**创建 Session 时**:
- [ ] 从用户消息提取关键词
- [ ] 创建规范的 session 目录名
- [ ] 创建并初始化 INDEX.md
- [ ] 通知用户 session 已创建

**执行任务时**:
- [ ] 所有临时文档保存在 session 目录下
- [ ] 及时更新 INDEX.md 的进度
- [ ] 重要决策记录在 INDEX.md 中
- [ ] 保持文档关系图更新

**完成任务时**:
- [ ] 更新 INDEX.md,进度改为 100%
- [ ] 检查所有改动的项目
- [ ] 记录改动清单
- [ ] 提示用户清理 session
