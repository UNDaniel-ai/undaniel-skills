# 1. 复杂度评分详解

## 概述

复杂度评分是 Complex Task Solver 的核心机制，通过 **5 因子模型**自动评估任务复杂度，并路由到合适的流程。

## 5 因子评分模型

### 因子 1: fileCount（文件数量）

**定义**：任务涉及的文件数量

| 文件数量 | 分数 | 说明 |
|---------|------|------|
| 1-5 个文件 | 0-2 分 | 简单任务，影响范围小 |
| 6-15 个文件 | 3-4 分 | 中等任务，涉及多个文件 |
| >15 个文件 | 5 分 | 复杂任务，大规模变更 |

**评分细则**：
- 1-2 个文件 → 0 分
- 3-5 个文件 → 2 分
- 6-10 个文件 → 3 分
- 11-15 个文件 → 4 分
- >15 个文件 → 5 分

**示例**：
- 修复单个组件样式 → 1 个文件 → 0 分
- 新增用户权限功能 → 8 个文件 → 3 分
- 重构数据层 → 40 个文件 → 5 分

---

### 因子 2: crossModule（跨模块影响）

**定义**：任务影响的模块或子系统数量

| 模块数量 | 分数 | 说明 |
|---------|------|------|
| 单模块 | 0-2 分 | 变更局限于一个模块 |
| 2-3 个模块 | 3-4 分 | 涉及多个模块协作 |
| >3 个模块 | 5 分 | 跨系统级别变更 |

**评分细则**：
- 单模块内部 → 0 分
- 单模块但影响接口 → 2 分
- 2 个模块 → 3 分
- 3 个模块 → 4 分
- >3 个模块 → 5 分

**示例**：
- 修改前端组件样式 → 单模块 → 0 分
- 新增 API 接口（前端+后端） → 2 个模块 → 3 分
- 数据层重构（前端+后端+数据库+缓存） → 4 个模块 → 5 分

---

### 因子 3: archChange（架构变更程度）

**定义**：任务对系统架构的影响程度

| 架构变更 | 分数 | 说明 |
|---------|------|------|
| 无架构变更 | 0-2 分 | 不涉及架构调整 |
| 局部架构调整 | 3-4 分 | 局部设计模式变更 |
| 全局架构重构 | 5 分 | 系统架构级别变更 |

**评分细则**：
- 无架构变更（仅实现细节） → 0 分
- 小调整（如：新增设计模式） → 2 分
- 局部架构调整（如：分层调整） → 3 分
- 大规模架构调整 → 4 分
- 全局架构重构 → 5 分

**示例**：
- 修复 bug → 无架构变更 → 0 分
- 新增数据缓存层 → 局部架构调整 → 3 分
- REST 迁移到 GraphQL → 全局架构重构 → 5 分

---

### 因子 4: ambiguity（需求清晰度）

**定义**：需求的模糊程度（分数越高越模糊）

| 清晰度 | 分数 | 说明 |
|--------|------|------|
| 需求明确清晰 | 0-2 分 | 实现路径清晰 |
| 部分需求模糊 | 3-4 分 | 需要讨论和澄清 |
| 高度模糊需求 | 5 分 | 需要探索和方案评估 |

**评分细则**：
- 需求完全明确，实现路径清晰 → 0 分
- 需求明确但有小疑问 → 2 分
- 需求部分模糊，需要讨论 → 3 分
- 需求多处模糊 → 4 分
- 需求高度模糊，需要 Brainstorm → 5 分

**判断标准**：
- 用户是否提供具体的验收标准？
- 实现路径是否有多种可能？
- 是否需要与用户多次对齐？

**示例**：
- "修改按钮颜色为 #0066cc" → 需求明确 → 0 分
- "优化登录流程" → 部分模糊 → 3 分
- "提升系统性能" → 高度模糊 → 5 分

---

### 因子 5: risk（技术风险等级）

**定义**：任务实施的技术风险

| 风险等级 | 分数 | 说明 |
|---------|------|------|
| 无明显风险 | 0-2 分 | 成熟技术，风险可控 |
| 可控风险 | 3-4 分 | 有一定挑战，但可控 |
| 高风险/未知领域 | 5 分 | 新技术或不确定因素多 |

**评分细则**：
- 无明显风险 → 0 分
- 小风险（如：性能影响） → 2 分
- 中等风险（如：兼容性问题） → 3 分
- 高风险（如：数据迁移） → 4 分
- 极高风险或未知领域 → 5 分

**风险评估维度**：
- **技术成熟度**：是否使用新技术或不熟悉的技术？
- **数据安全**：是否涉及数据迁移或删除？
- **兼容性**：是否可能破坏现有功能？
- **回滚难度**：出问题后是否容易回滚？

**示例**：
- 修改样式 → 无明显风险 → 0 分
- 新增权限功能 → 可控风险 → 3 分
- 数据库表结构重构 → 高风险 → 5 分

---

## 综合评分计算

### 计算公式

```
综合评分 = (fileCount + crossModule + archChange + ambiguity + risk) / 5
```

**说明**：
- 每个因子满分 5 分
- 总分满分 25 分
- 综合评分范围：0-10 分（除以因子数量）

### 评分示例

#### 示例 1：简单任务

**任务**："修复登录页面按钮颜色不一致"

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 0 | 1 个文件 |
| crossModule | 0 | 单模块 |
| archChange | 0 | 无架构变更 |
| ambiguity | 1 | 需求基本明确 |
| risk | 0 | 无明显风险 |
| **综合评分** | **0.2** | 路由到 Route A |

---

#### 示例 2：中等任务

**任务**："新增用户权限管理功能"

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 3 | 8 个文件 |
| crossModule | 3 | 前端+后端 |
| archChange | 1 | 小调整（新增权限模块） |
| ambiguity | 2 | 需要确认角色和权限粒度 |
| risk | 2 | 需要注意权限校验逻辑 |
| **综合评分** | **2.2** | 路由到 Route B |

---

#### 示例 3：复杂任务

**任务**："REST API 迁移到 GraphQL"

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 5 | 40+ 个文件 |
| crossModule | 5 | 前端+后端+数据库+缓存 |
| archChange | 5 | 全局架构重构 |
| ambiguity | 3 | 需要确认迁移策略 |
| risk | 4 | 数据迁移和兼容性风险 |
| **综合评分** | **4.4** | 路由到 Route C |

---

## 路由决策

### 评分阈值

| 综合评分 | 路由 | 流程 |
|---------|------|------|
| **< 3.0 分** | Route A | 快速流程 |
| **3.0-6.0 分** | Route B | 标准流程 |
| **≥ 6.0 分** | Route C | 完整流程 |

### 边界案例处理

**情况 1：评分接近阈值（如 2.8 或 3.2）**
- 向用户展示评分结果
- 说明建议流程和理由
- 允许用户手动调整流程级别

**情况 2：评分明显偏离实际复杂度**
- AI 主动询问用户是否调整
- 记录调整理由，用于模型优化

**情况 3：用户明确指定流程**
- 优先遵循用户指定
- 但仍展示评分结果，供用户参考

## 评分透明化

### 展示评分结果

向用户展示评分结果时，使用以下格式：

```markdown
## 复杂度评估

我对这个任务进行了复杂度评估：

| 因子 | 分数 | 说明 |
|------|------|------|
| fileCount | 3 | 涉及 8 个文件 |
| crossModule | 3 | 跨前端和后端 2 个模块 |
| archChange | 1 | 需要新增权限模块（小调整） |
| ambiguity | 2 | 需要确认角色类型和权限粒度 |
| risk | 2 | 需要注意权限校验的安全性 |

**综合评分**：2.2 分

**建议流程**：Route B（标准流程）

这个任务属于中等复杂度，我会进行需求对齐、技术方案讨论、设计输出、影响分析、实现计划，然后执行。

如果你认为应该使用更简化或更完整的流程，请告诉我。
```

## 最佳实践

1. **评分前先理解任务**：仔细阅读用户需求，必要时提问澄清
2. **多维度考虑**：不要只看文件数量，综合考虑所有因子
3. **留有调整空间**：评分接近阈值时，询问用户意见
4. **记录评分理由**：方便后续回顾和模型优化
5. **动态调整**：实施过程中发现评分不准确，及时调整

## 常见问题

### Q1: 评分偏低但实际很复杂怎么办？

**A**: 实施过程中发现复杂度高于预期时：
1. 立即告知用户评分可能不准确
2. 建议升级到更高级别的流程
3. 记录调整理由，用于模型优化

### Q2: 用户坚持使用不匹配的流程？

**A**: 尊重用户选择，但：
1. 提醒可能的风险（如：简单流程可能遗漏重要设计）
2. 记录用户选择理由
3. 实施过程中灵活调整

### Q3: 多个因子评分差异很大？

**A**: 这是正常现象，例如：
- 文件少但架构变更大 → 可能是架构重构
- 文件多但风险小 → 可能是批量修改

综合评分会平衡各个因子，给出合理的路由决策。

## 评分校准

### 定期校准

建议定期回顾评分准确性：

1. **收集实际案例**：记录评分和实际执行情况
2. **分析偏差**：哪些因子经常偏差？
3. **调整阈值**：根据团队习惯调整评分标准
4. **更新模型**：优化评分公式和权重

### 团队协作

如果团队使用 Complex Task Solver：

- 统一评分标准
- 共享校准结果
- 定期讨论边界案例
