# 1. 复杂度评分详解

## 概述

复杂度评分用于决定任务走 Route A / B / C。评分不是为了“精确建模”，而是为了防止复杂任务被当作简单任务直接开做。

## 5 因子评分模型

每个因子分值范围都是 `0-5`，综合评分取平均值，因此最终分数范围固定为 `0.0-5.0`。

| 因子 | 说明 | 评分提示 |
|------|------|----------|
| `fileCount` | 涉及文件数量 | 文件越多，分数越高 |
| `crossModule` | 跨模块影响 | 影响模块越多，分数越高 |
| `archChange` | 架构变更 | 从实现细节到全局重构递增 |
| `ambiguity` | 需求模糊度 | 需要讨论和方案评估时升高 |
| `risk` | 技术与交付风险 | 数据迁移、兼容性、回滚困难时升高 |

### 评分参考

#### fileCount

- `0`：1-2 个文件
- `2`：3-5 个文件
- `3`：6-10 个文件
- `4`：11-15 个文件
- `5`：>15 个文件

#### crossModule

- `0`：单模块内部变更
- `2`：单模块但影响接口
- `3`：2 个模块
- `4`：3 个模块
- `5`：>3 个模块

#### archChange

- `0`：无架构变更
- `2`：小型模式调整
- `3`：局部分层或链路调整
- `4`：大规模结构调整
- `5`：全局架构重构或技术栈迁移

#### ambiguity

- `0`：需求和验收标准都明确
- `2`：需求明确，但有少量待确认问题
- `3`：需要一轮正式需求对齐
- `4`：需要多轮讨论才能锁定方案
- `5`：需要 Brainstorm 多方案评估

#### risk

- `0`：成熟方案、回滚容易
- `2`：存在局部风险，但容易控制
- `3`：兼容性或回归风险明显
- `4`：迁移 / 数据 / 长链路协作风险高
- `5`：高不确定性或回滚困难

## 路由阈值

| 综合评分 | 路由 | 说明 |
|---------|------|------|
| `0.0-1.9` | Route A | 简单任务，轻量处理 |
| `2.0-3.4` | Route B | 中等任务，必须走需求/设计/计划门禁 |
| `>= 3.5` | Route C | 复杂任务，必须走完整流程与 Brainstorm |

## 强制升级条件

即使平均分不高，只要满足以下条件之一，也要升级：

- 存在跨模块 / 跨仓 / 多阶段链路
- 需要阶段里程碑或跨 Chat 恢复
- 需要正式设计文档后才能落地
- 存在高风险迁移、架构重构或多方案竞争

## 评分示例

### 示例 1：简单任务

任务：修复登录页面按钮颜色不一致

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 0 | 1 个文件 |
| crossModule | 0 | 单模块 |
| archChange | 0 | 无架构变更 |
| ambiguity | 1 | 需求基本明确 |
| risk | 0 | 无明显风险 |
| **综合评分** | **0.2** | Route A |

### 示例 2：中等任务

任务：新增用户权限管理功能

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 3 | 8 个文件 |
| crossModule | 3 | 前端 + 后端 |
| archChange | 2 | 引入新的权限链路 |
| ambiguity | 3 | 需要确认角色粒度和权限模型 |
| risk | 3 | 回归和权限校验风险 |
| **综合评分** | **2.8** | Route B |

### 示例 3：复杂任务

任务：REST API 迁移到 GraphQL

| 因子 | 分数 | 理由 |
|------|------|------|
| fileCount | 5 | 40+ 个文件 |
| crossModule | 5 | 前端 + 后端 + 数据 + 缓存 |
| archChange | 5 | 全局架构迁移 |
| ambiguity | 3 | 需要确认迁移策略 |
| risk | 4 | 兼容性与回滚风险高 |
| **综合评分** | **4.4** | Route C |

## 边界案例处理

- 分数接近阈值时，默认向上取整到更保守的流程
- 用户若坚持更轻流程，必须显式记录风险
- 实施过程中发现复杂度升级时，应升级流程，而不是继续硬做

## 与后续流程的关系

- Route A：只需要最小需求确认与验证
- Route B：进入硬门禁协议
- Route C：进入硬门禁协议，并强制 Brainstorm
