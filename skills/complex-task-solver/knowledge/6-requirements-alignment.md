# 6. 需求对齐流程

## 概述

需求对齐是确保 AI 充分理解用户需求的关键阶段，适用于 Route B 和 Route C。

## 5 维判断清单

### 维度 1: 目标清晰度

**问题**：用户的目标是否明确？

| 情况 | 处理方式 |
|------|----------|
| ✅ 目标明确 | 确认目标，继续下一维度 |
| ❌ 目标模糊 | 提问澄清，如："你想实现什么效果？" |

**检查清单**：
- [ ] 用户想要达成什么目标？
- [ ] 这个目标是否可衡量？
- [ ] 是否有明确的成功标准？

---

### 维度 2: 范围明确性

**问题**：任务的边界是否清楚？

| 情况 | 处理方式 |
|------|----------|
| ✅ 范围明确 | 确认范围，继续下一维度 |
| ❌ 范围模糊 | 提问澄清，如："是否包括 X 功能？" |

**检查清单**：
- [ ] 哪些内容在范围内？
- [ ] 哪些内容在范围外？
- [ ] 是否有边界案例需要讨论？

---

### 维度 3: 约束条件

**问题**：有哪些限制条件？

| 约束类型 | 示例 |
|---------|------|
| 技术约束 | 必须使用特定技术栈 |
| 时间约束 | 必须在某个时间点前完成 |
| 资源约束 | 不能引入新的依赖 |
| 兼容性约束 | 必须向后兼容 |

**检查清单**：
- [ ] 是否有技术限制？
- [ ] 是否有时间限制？
- [ ] 是否有资源限制？
- [ ] 是否有兼容性要求？

---

### 维度 4: 优先级

**问题**：什么最重要？

| 优先级 | 说明 |
|--------|------|
| P0 | 核心功能，必须实现 |
| P1 | 重要功能，应该实现 |
| P2 | 次要功能，可选实现 |

**检查清单**：
- [ ] 核心功能是什么？
- [ ] 哪些功能可以后续优化？
- [ ] 如果时间不够，可以放弃哪些功能？

---

### 维度 5: 验收标准

**问题**：如何判断任务完成？

**检查清单**：
- [ ] 有哪些具体的验收标准？
- [ ] 如何验证功能正常？
- [ ] 是否需要测试用例？

---

## 需求对齐模板

```markdown
## 需求对齐

### 核心需求
[用 1-2 句话描述核心需求]

### 详细说明

**What（是什么）**：
- [要实现什么功能]

**Why（为什么）**：
- [为什么需要这个功能]

**Who（谁使用）**：
- [谁会使用这个功能]

**When（何时触发）**：
- [什么时候会用到]

**Where（在哪里）**：
- [在系统的哪个位置]

**How（如何实现）**：
- [初步的实现思路]

### 约束条件
- [约束 1]
- [约束 2]

### 优先级
- **P0（核心）**：[核心功能]
- **P1（重要）**：[重要功能]
- **P2（可选）**：[次要功能]

### 验收标准
1. [标准 1]
2. [标准 2]
3. [标准 3]

---

我的理解是否准确？是否需要补充或调整？
```

## 示例

**用户需求**："优化系统性能"

**AI 输出**：

```markdown
## 需求对齐

### 核心需求
优化系统性能，提升用户体验。

### 详细说明

**What（是什么）**：
- 请问具体想优化哪方面的性能？
  - 页面加载速度？
  - API 响应时间？
  - 数据库查询性能？
  - 其他？

**Why（为什么）**：
- 当前遇到了什么性能问题？
- 用户是否有反馈？

**Who（谁使用）**：
- 哪些用户受影响最大？
- 全部用户还是特定场景？

**When（何时触发）**：
- 什么场景下性能问题明显？
- 高峰期还是特定操作？

**Where（在哪里）**：
- 性能瓶颈在系统的哪个部分？

**How（如何实现）**：
- 有没有初步的优化方向？

### 约束条件
- 是否有时间限制？
- 是否可以修改架构？
- 是否可以引入新技术（如缓存）？

### 优先级
- 请按重要性排序想优化的内容

### 验收标准
- 期望优化到什么程度？
  - 页面加载时间从 X 秒降到 Y 秒？
  - API 响应时间从 X ms 降到 Y ms？

---

请提供更多细节，帮助我更好地理解你的需求。
```

## 复杂任务分解触发条件

在需求对齐过程中，如果发现以下情况，应考虑任务分解：

| 触发条件 | 说明 |
|---------|------|
| 涉及多个模块 | 需要拆分为独立任务 |
| 实施周期长（>2 小时） | 需要拆分为可追踪的步骤 |
| 存在并行机会 | 识别可并行执行的部分 |
| 依赖关系复杂 | 需要明确任务依赖 |

---

## 最佳实践

1. **主动提问**：需求模糊时主动提问，不要猜测
2. **确认理解**：用自己的话复述需求，请用户确认
3. **识别边界**：明确哪些在范围内，哪些不在
4. **记录约束**：记录所有约束条件，避免后期返工
5. **动态调整**：实施过程中发现新需求，及时对齐

---

## 常见问题

### Q1: 用户需求极度模糊怎么办？

**A**: 通过提问引导用户：
- 从具体场景入手："能否描述一个具体的使用场景？"
- 从问题入手："当前遇到了什么问题？"
- 从目标入手："你希望达成什么效果？"

---

### Q2: 用户需求频繁变化怎么办？

**A**:
- 每次变化都重新进行需求对齐
- 评估变化对实现计划的影响
- 必要时重新评估复杂度和路由决策

---

### Q3: 如何判断需求对齐是否充分？

**A**: 检查以下条件：
- ✅ 5 维判断清单全部确认
- ✅ 验收标准明确且可衡量
- ✅ AI 和用户对需求理解一致
- ✅ 约束条件全部记录
